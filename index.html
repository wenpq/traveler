FRIB Traveler

A Web application to create, manage, and share forms, travelers, and working packages

A traveler contains a sequence of actions defined by a user in a form. An action requires a person to verify the action is performed and/or collect data as defined in the form. The traveler application records who collects what data at what time before the traveler is completed.

Login and logout

A user can log into the the application by her/his FRIB windows username and password.

Interface layout

The application user interface is composed of top navigation bar, top buttons, and tabs. Each tab contains its own buttons and a data table. The top buttons link to the device list page, user profile page, and also the user manual page.

Data table

The drop down input on the left top of the table controls how many entries are displayed in the current page. The default number is 10.

The left bottom of the table shows the current displayed entries of the total entries.

The right bottom shows the pagination of the view.

The Search input on the top right of a table can be used to filter all the searchable columns in a table. Only the matched rows will be displayed when the user is typing. No matching records found will be displayed if no match found.

Most columns in a table have a text input at the bottom that can be used to filter the corresponding column. The column filter inputs and the top search input can be combined together.

Most columns in a data is sortable. A column is sorted when a user clicks on the header of the column. A small blue triangle shows the current sorting direction.

A user can use the buttons on the top left to copy the entries to the clipboard, or print, or export it to spreadsheet or pdf file formats. Note that only the filtered results will be copied, printed, or exported. This feature can be used to generate reports.

Compose a form

The What-You-See-Is-What-You-Get form builder supports basic input types include checkbox, text, number, paragraph (textarea) and file upload. It also supports other HTML5 input types like date, date time, time, email, phone number and URL. Note that not all browsers natively support these types. In addition, the builder provides two advanced types: rich instruction and section. The section type will result a legend header for section titles, and a side navigation panel will be generated for forms with more than one section. The rich instruction can be used for instruction content with special characters, formats, pictures and links. When a form is saved, the application will assign the form a unique id. A user can save an existing form as a new form with different name, and starts to work on it. This can be used for form version control.




Validation rules

The What-You-See-Is-What-You-Get form builder also supports basic validation rules like required and min/max for a number input. This is a new feature released in v2.2.0. Basically, to make an input required, check the "required" checkbox in the input's specification. The min/max rule works in the same way. A user can test the validation rules of a form in its preview before creating a traveler based on it. The validation messages can be shown or refreshed by clicking on the "Show validation" button. The validation works similarly in a traveler page.

Share a form

A form's owner can share it with other users. A shared form will appear in other user's "Shared form" tab. A user with write permission can revise the shared form, and create a clone of her/his own based on it. A user with read permission can only view the form. The form owner can add/remove users in the share list and modify the users' permissions.

Create a traveler

A user can create a new traveler from a owned form. The traveler will appear in the "My travelers" and "All travelers" tabs.

Configure a traveler

A traveler owner can specify a traveler's name, description, deadline, and devices on the config page. A traveler's status can also be modified on the config page. When a traveler is created, its status is "initialized". When starting to work, its status turns to "active". The traveler can collect data when it is active. The owner can freeze a traveler in order to suspend the traveler, and resume the work before the traveler is completed. A frozen traveler cannot collect data.

Share a traveler

To share a traveler is similar to share a form. A user with write permission can update the data in the traveler, while a user with read permission can only view the traveler with collected data. Any user can view the traveler with a link or in the "All travelers" tab.

Work on a traveler

The traveler owner and other users with write permission can update the data in a traveler. All data input history are kept and shown on the traveler page. The progress can be seen in the traveler data table. A user can request to complete a traveler. The traveler owner can approve the completion, or asks for more work.

Clone a traveler

An active or complete traveler can be cloned by its owner or a user having write permission. When a traveler is cloned, the new traveler will have the same form, title/description, and shared user list as the source. The device list will be empty. The cloned traveler is active by default.

Archive a traveler

When a traveler is not relevent any longer, the owner can choose to archive it in order to clean the traveler list and present accident usage. An archived travler will be only visible to its owner. An archived traveler can also be de-archived.
